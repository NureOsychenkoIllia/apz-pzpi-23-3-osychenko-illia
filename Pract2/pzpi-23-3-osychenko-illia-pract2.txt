                         3 ОПИС ВИКОНАНОЇ РОБОТИ

     3.1 Загальні відомості про систему

     Назва системи: Visual Studio Code.
     Розробник: Microsoft.
     Призначення системи: Visual Studio Code — це кросплатформне середовище
редагування та розробки програмного забезпечення. Система поєднує редактор
коду, роботу з проєктами, інтеграцію з Git, термінал, налагодження, підтримку мов
програмування через розширення та віддалені сценарії розробки.
     VS Code є складною програмною системою, оскільки він працює в кількох
середовищах: як настільний застосунок на Electron, як вебредактор у браузері
та як клієнт до віддалених середовищ на базі SSH, WSL, контейнерів або
GitHub Codespaces. Його функціональність значною мірою розширюється через
Marketplace і Extension API.

     3.2 Архітектурний стиль системи

     Архітектуру    VS    Code   доцільно    розглядати   як   поєднання    кількох
архітектурних стилів.
     Модульна шарова архітектура. Офіційна організація коду описує ядро VS
Code як шарову систему. Основні шари: «base» , «platform» , «editor» , «workbench» ,
«code» і «server» . Нижні шари містять загальні утиліти й сервіси, а верхні формують
редактор, робоче середовище та точки входу настільної або серверної частини.
     Подієво-орієнтована архітектура. Команди, події активації розширень, зміни
документів, повідомлення від мовних серверів і debug-сесій обробляються
асинхронно. Це дозволяє запускати функціональність лише тоді, коли вона потрібна
користувачу.
     Клієнт-серверна та протокольна взаємодія. Language Server Protocol (LSP) і
Debug Adapter Protocol (DAP) дають змогу винести мовний аналіз і налагодження
в окремі процеси. Це зменшує зв’язаність і дозволяє підтримувати багато мов без
переписування ядра системи.

     3.3 Основні компоненти системи

     Основні компоненти VS Code подано на рисунку 3.1.


                          Загальна архітектура VS Code

                          →                             ↔
      Main process             Renderer            /         Extension Host
   життєвий цикл, OS,              Workbench                    розширення
             CLI               UI, вкладки, панелі


                          ↔                             ↔
      Monaco Editor             Language Servers            Debug Adapters
      модель тексту                      LSP                       DAP

                   Рисунок 3.1 – Загальна архітектура VS Code

     Головний процес (Main process) відповідає за життєвий цикл застосунку,
створення вікон, взаємодію з операційною системою та роботу CLI.
     Рендерер-процес (Renderer / Workbench) формує вікно користувача та робоче
середовище: вкладки, Explorer, статус-бар, термінал тощо. Такий розподіл процесів
важливий для надійності: потенційно нестабільна робота не повинна блокувати
інтерфейс.
     Шар Base містить загальні утиліти та базові блоки UI, незалежні від логіки
редактора.
     Шар Platform визначає сервісну архітектуру, реєстрацію сервісів та механізм
впровадження залежностей (dependency injection).
     Шар Editor (Monaco Editor Core) відповідає за модель тексту, редагування,
виділення та базові мовні можливості.
     Шар Workbench збирає редактор у повноцінне середовище розробки з
експлорером, панелями, Git та дебагом.
     Extension Host запускає розширення в окремому процесі, надаючи їм доступ
до API редактора. Це запобігає блокуванню UI сторонніми кодами та дозволяє
«ліниву» активацію за подіями ( «activationEvents» ).
                                                                                5
     Language Servers та Debug Adapters винесені в протокольну модель (LSP/
DAP), що дозволяє мовним серверам бути написаними будь-якою мовою та не
навантажувати UI-процес.

     3.4 Взаємодія компонентів

     Взаємодія компонентів адаптується до середовища розміщення:
      –   У Desktop режимі UI та Workspace знаходяться на одній машині.
      –   У Віддаленому режимі UI залишається локальним, а Workspace і частина
          розширень працюють у WSL, Docker-контейнері або на віддаленому
          сервері через SSH.
      –   У Браузерному режимі система працює через WebWorker Runtime з
          обмеженим доступом до Node.js API.
     Сервісна модель та поділ точок входу ( «code» , «server» ) дозволяють
повторно використовувати спільний код у всіх цих сценаріях без дублювання.

     3.5 Масштабованість та надійність

     Масштабованість VS Code проявляється через екосистему та навантаження
між процесами:
     Розширюваність через API. Ядро не містить усіх функцій; вони додаються
через розширення за стабільними контрактами.
     Лінива активація (Lazy Activation). Розширення активуються лише за потреби
(наприклад, при відкритті Markdown-файлу), що економить ресурси CPU та пам’яті.
     Ізоляція процесів. UI, Extension Host, мовні сервери та адаптери налагодження
працюють окремо. Нестабільний аналіз коду мінімально впливає на головний
інтерфейс.
     Поділ середовищ. Реалізація сервісів адаптується під desktop, web або remote,
зберігаючи спільне логічне ядро.

     3.6 Приклади програмної реалізації

     Перший приклад демонструє, як розширення описує події активації, entry
points для Node.js і браузера, а також бажане місце виконання:
                                                                          6
      1 {
      2   "activationEvents": [
      3      "onLanguage:markdown",
      4      "onCommand:architectureDemo.showInfo"
      5   ],
      6 "main": "./dist/node/extension.js",
      7 "browser": "./dist/web/extension.js",
      8 "extensionKind": ["workspace", "ui"]
      9 }


      Другий приклад демонструє запуск language client, який створює окремий
language server і спілкується з ним через IPC:

      1 const serverOptions = {
      2 run: { module: serverModule, transport: TransportKind.ipc },
      3 debug: { module: serverModule, transport: TransportKind.ipc }
      4 };
      5
      6 client = new LanguageClient(
      7 "architectureDemo",
      8 "Architecture Demo Language Server",
      8 serverOptions,
      9 clientOptions
      10 );
      11
      12 await client.start();

        Основні результати аналізу:
        –   Архітектура VS Code базується на розділенні обов’язків між процесами
            (Main, Renderer, Extension Host), що забезпечує стабільність інтерфейсу.
        –   Внутрішня структура коду організована за шарами ( «base» , «platform» ,
            «editor» , «workbench» ), що дозволяє створювати стабільні архітектурні
            контракти та забезпечувати гнучкість платформи.
        –   Використання протоколів LSP та DAP дозволяє масштабувати підтримку
            мов програмування та налагоджувачів без ускладнення ядра системи.
        –   Система підтримує різні середовища (Desktop, Web, Remote) завдяки
            модульній сервісній архітектурні та впровадженню залежностей.
        Отже, VS Code є прикладом сучасної, масштабованої та надійної програмної
архітектури, яка ефективно балансує між продуктивністю та розширюваністю.

TypeScript-коду, де створюється «LanguageClient» , задаються «serverOptions» із
транспортом IPC і викликається «client.start()» для запуску окремого language server.
      Уточнення:
      «Як пояснити ці приклади у звіті, щоб вони були пов’язані саме з
архітектурою, а не виглядали як довільні фрагменти коду?»
      Відповідь: Manifest розширення слід пояснювати як опис архітектурного
контракту між ядром VS Code та розширенням: події активації визначають, коли
код буде завантажено, а «main» , «browser» і «extensionKind» визначають можливі
runtime та місце виконання. Фрагмент Language Client демонструє інший контракт
— взаємодію з окремим процесом мовного сервера через LSP. Обидва приклади
показують, що VS Code масштабує функціональність через протоколи й API, а не
через додавання всієї логіки до ядра редактора.
